ファイルサーバー

現在のファイルサーバーは2025年11月末に保守が終了します。保守終了後は、ハードウェア故障時の部品調達不可・ベンダーサポート終了により、ファイルサーバーが停止した場合、全社のファイル共有・業務データへのアクセスが不可能となり、業務継続に致命的な影響を与えるリスクを抱えています。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
当面の致命的なリスクを回避するため、ファイルサーバーの保守延長も検討している。 現在のインフラは以下の深刻な課題を抱えています。保守延長では一時的な延命措置となり根本的な課題解決に至りません。
①必要最低限の物理機器構成：冗長性が不足し、単一障害点が多数存在
②ブラックボックス化したネットワーク構成：ドキュメント不足により、トラブル時の対応が困難で復旧に長時間を要する
③監視体制の不備：障害の早期発見ができず、業務影響が拡大するリスク
④保守コストの増大：延長保守料金は通常保守の1.5～2倍に跳ね上がり、年数経過とともにさらに高額化
⑤非機能要件の未達成：可用性（99.9%のSLA）、バックアップ（RPO：1日、RTO：4時間）、セキュリティ要件（暗号化、アクセス制御）などの要求水準を満たせない
これらの課題を総合的に解決するため、保守延長ではなくクラウド移行（Amazon FSx for Windows File Server）を強く推奨します。クラウド化により、高可用性、自動バックアップ、監視体制の確立、コスト予測可能性、ドキュメント化された構成管理が実現できます。
提案されている統合基盤では、Amazon FSx for Windows File Server（フルマネージドSMBファイルサーバー）を活用することが説明された。 Amazon FSx for Windows File Serverは、AWSのフルマネージドSMBファイルサーバーであり、①SMB/CIFSプロトコル完全サポート（既存アプリケーションとの互換性）、②NTFS ACLによる詳細な権限設定とAD統合、③マルチAZ構成による自動フェイルオーバー、④AWS Backup統合による自動バックアップ・ポイントインタイムリカバリ、⑤ストレージ容量の動的拡張（ダウンタイムなし）、⑥OSパッチ適用の自動化、⑦CloudWatch統合による監視・アラート、という技術的優位性があります。オンプレミスと比較して、可用性・セキュリティ・運用性が向上し、ハードウェア保守からの解放・コスト最適化（従量課金、初期投資削減）を実現できるため、提案しています。
Dropboxなどのクラウドストレージサービスでは、フォルダ単位の共有は可能だが、Windowsのファイルシステムとは技術的に異なるため、AccessやExcelマクロ、CSV連携といった既存業務アプリケーションが動作不能となる可能性が高いという業務断絶リスクが議論された。特に、DropboxのファイルシステムはWindowsのファイルシステムとは違うため、その上でAccessのマクロやプログラムは動作しないことが技術的な根拠として説明された。 Dropbox等のクラウドストレージは同期型アーキテクチャ（ローカルにダウンロード→編集→アップロード）であり、Windows環境の従来型ファイル共有（SMB/CIFSプロトコル）とは技術アーキテクチャが根本的に異なります。特に、①Access共有データベースは、複数ユーザーが同時にSMB経由でファイルを開き、レコードレベルのロック（.laccdb/.ldbファイル）を使用しますが、Dropbox同期ではファイルロック機構が正常に動作せず、データ破損・競合が発生します。②UNCパス依存（\\server\share\data.csv）のExcelマクロ・バッチスクリプトが、Dropboxのローカルフォルダパス（C:\Users\\Dropbox\...）では動作しません。③リアルタイム性が必要な業務（基幹システムがCSV出力→Excelマクロが即座に処理）において、Dropbox同期のタイムラグが業務に影響します。そのため、SMB/CIFSプロトコルをサポートするAmazon FSx for Windows File Serverを提案しています。

ネットワーク

現在のVPN関連サービスは2026年3月に富士通のサポートが終了します。サポート終了後は、VPN機器の故障や設定変更が必要な場合の対応が不可能となり、各拠点から本社システムへの接続ができなくなる事態が発生し、拠点業務が停止するリスクを抱えています。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
VPN関連サービスは2026年3月に富士通のサポートが終了するという状況が共有された。 VPNがない場合、①営業所から本社サーバー・システムへのL3接続が不可、②ファイルサーバー（SMB）、業務システムへのアクセス不可、③代替手段（VPNなしでインターネット経由アクセス）はセキュリティリスク（平文通信、認証の脆弱性）が高い、という業務断絶リスクが発生します。
富士通のサポートは終了し、ソフトバンクによる継続契約は可能であるという状況が共有された。 ファイルサーバーと同様にソフトバンクによる継続契約では根本的な解決にはなりません。現在のVPN構成には以下の課題があります。
①監視体制の欠如：VPN機器の死活監視・トラフィック監視がなく、障害発生時の検知が遅れる
②保守コストの増大：継続契約の保守費用は年々上昇し、機器老朽化により障害頻度も増加
③運用管理の複雑性：設定変更には専門知識が必要で、ドキュメント不足により属人化
④非機能要件の未達成：可用性（99.9%）、セキュリティ（最新の暗号化方式）、拡張性（拠点追加時の柔軟な対応）などの要件を満たせない
ファイルサーバーのAWS移行に合わせて、AWS Site-to-Site VPNを採用することを強く推奨します。AWSマネージドサービスにより、①CloudWatch統合による24時間365日の自動監視、②予測可能な月額料金（従量課金）、③管理画面での簡単な設定変更、④99.95%のSLA保証と最新セキュリティ基準への自動対応が実現でき、非機能要件を完全に満たすことができます。
クラウド移行は、単なる「現状の置き換え」ではなく、「当面の維持・安全化」を優先し、将来的には簡素で運用容易な形へ段階的に最適化していく考え方を共有した。 クラウド移行は単なる「現状の置き換え」ではなく、当面の維持・安全化を優先します。保守終了リスクを回避し、既存業務を継続するため、AWS Managed Microsoft AD、Amazon FSx for Windows File Server、AWS Site-to-Site VPNにリフト&シフトすることを提案しています。将来的には、ネットワーク構成の簡素化やコスト最適化など、その時点での技術動向や業務要件に応じた最適な方式に柔軟に対応できます。
現状のネットワーク構成について、PC約120台という規模に対してCiscoの上位L3機器が実質使われておらず、規模に合わない構成になっているため、ネットワークの簡素化（フラットな構成、L3機器の削減、回線の見直し）を行い、ランニングコストを低減すべきという要望が出た。 現状のネットワーク構成は、PC約120台という規模に対して、Ciscoの上位L3機器（高性能なルーター/スイッチ）が導入されているものの、その高度な機能（VLAN間ルーティング、QoS制御、マルチキャスト等）が実質的に活用されておらず、規模に合わない過剰な構成になっています。これにより、①高額な機器コスト、②複雑な設定による運用負荷、③トラブルシューティングの困難化、というコストと運用の非効率が発生しています。そのため、AWS上でネットワークをフラットな構成に再設計することを提案しています。具体的には、①AWS Site-to-Site VPN（各拠点との安全なIPsec接続）により、オンプレミスの複雑なVPN機器を不要にし、②AWS Transit Gateway (TGW)（ネットワークハブ機能）により、拠点間接続をハブ&スポーク型のシンプルな構成で実現します。これにより、不要な高機能L3機器を削減し、回線の見直し（IP-VPN集中接続化等）を行い、ランニングコストを低減できます。

AD (Active Directory)

現在のADサーバーは2025年11月以降に保守が終了予定です。保守終了後は、ハードウェア故障時の部品調達やベンダーサポートが受けられなくなり、ドメインコントローラーが停止した場合、全社のWindows認証・ファイルアクセス・PC管理が不可能となる重大な業務停止リスクを抱えています。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
ADが必要であるか否かについて、「ファイルサーバーの権限管理やキッティング効率化に有効である」という見解が示された。 ADは単なるファイルサーバーの権限管理だけでなく、①統合認証（Kerberos）によるシングルサインオン、②グループポリシー（GPO）によるPC120台の一括セキュリティ設定（パスワードポリシー、セキュリティソフト導入、USBメモリ使用禁止等の統一設定）、③LDAP連携による他システムとの統合、④組織階層に基づくアクセス制御（RBAC）という4つの重要な機能を提供します。特にGPOは、組織単位（OU）ごとに統一されたセキュリティ設定を強制適用でき、設定ミスやセキュリティレベルの不統一のリスクを防ぎます。これらはWindows環境における標準的なエンタープライズIT基盤であり、相互に統合されています。AD廃止は技術的に現実的ではないため、継続利用を提案しています。
現在ADはAzureクラウドで管理されているのうではないかという質問を受けた。 現状はオンプレミスADとAzure ADで分断しており、以下の課題を抱えています。
【現状の分断構成】
・オンプレミスAD：WindowsクライアントPCとファイルサーバーの管理（グループポリシー、NTFS ACL）
・Azure AD（Microsoft Entra ID）：Microsoft 365とネクストセット認証
【発生している課題（部分最適の罠）】
①認証管理の分散化：現在、社員は「Windows PC用のパスワード」「Microsoft 365用のパスワード」「ネクストセット用のパスワード」など、システムごとに異なるパスワードを管理する必要があり、結果として同じパスワードを使い回すセキュリティリスクが増大
②運用負荷の爆発的増加：人事異動時に10個以上のシステムで個別権限変更（年間20名の異動で200回の手動作業）
③セキュリティリスク：例えば、退職者が出た際、「オンプレミスADでは削除したが、Azure ADでは削除し忘れた」という事態が発生すると、退職者がMicrosoft 365やネクストセットに継続アクセス可能となり、機密情報漏洩の重大リスクに
④Azure AD Connectの複雑な同期運用：オンプレミスADとAzure ADを同期させるAzure AD Connectは、同期エラー発生時の原因究明が困難で、「パスワード変更が反映されない」「新規ユーザーが同期されない」などの障害対応に専門知識が必要
AD、ファイルサーバー、VPNをAWS環境に統合することで、監視・バックアップ・セキュリティ管理を一元化し、これらの課題を解決することを提案しています。
オンプレミスADの保守終了リスクへの対応として、AWS Managed Microsoft ADを採用するメリットは何か。 オンプレミスADサーバーを継続利用する場合、冗長構成としてサブのドメインコントローラー（DC）が最低1台必要ですが、既存DCの保守が2025年11月以降に終了し、業務停止に直結する重大なリスクとなります。AWS Managed Microsoft ADへの移行により、以下のメリットが実現できます。
①ハードウェア保守からの解放・自動フェイルオーバー・ストレージの自動レプリケーション
②ドメインコントローラーのパッチ適用・バックアップ・ヘルスチェックがAWSにより自動化され、運用負荷が大幅に削減
③マルチAZ構成による高可用性が標準で提供され、ハードウェア故障時の自動フェイルオーバーにより、業務停止リスクが最小化
④構成や運用ドキュメントの公開による他社・自社での引き継ぎ可能性（ベンダーロックイン回避）
⑤スケーラビリティが高く、ユーザー数やアクセス負荷の増加に応じて柔軟に拡張可能
⑥AWS他サービス（FSx、ネットワーク系サービス等）との統合が容易で、統一されたID管理基盤を構築
⑦既存WindowsServerライセンスのBYOL継続利用が可能
これらにより、コスト削減・運用負荷削減・透明性確保・可用性向上を同時に実現できるため、採用を提案しています。
ライセンスは踏襲できる可能性がある。 既存のWindowsServerライセンスをBYOL（Bring Your Own License）として持ち込み、EC2上に独自のADを構築する選択肢があります。AWS Managed Microsoft ADは、Microsoftとの正式なライセンス契約に基づいて提供されており、ライセンスコンプライアンスの観点でも安全です。既存のオンプレミスADのドメイン構成、グループポリシー、ユーザー・グループ情報などを移行ツール（AWS Directory Service）を用いて継承可能であり、設定の踏襲が可能です。このため、ライセンスコストを抑えつつ、既存のAD環境を継続利用できます。

結論：ファイルサーバーを起点とした統合的なクラウド移行の必要性

ファイルサーバーの保守終了（2025年11月末）というクリティカルな課題を契機として、技術的に密接に連携するネットワークとADも同時にAWSへ移行することで、根本的な課題解決と業務継続性を確保できます。

保守延長ではなくクラウド移行を推奨する理由

  • 保守終了リスクの一括解消：ファイルサーバー（2025年11月）、AD（2025年11月以降）、VPN（2026年3月）の保守終了リスクを統合的に解決
  • ブラックボックス化の解消：ドキュメント化された構成管理により、透明性とベンダーロックイン回避を実現
  • 非機能要件の達成：可用性（99.9%→99.95%）、監視体制（24時間365日自動監視）、セキュリティ（最新基準への自動対応）
  • 運用負荷とコストの削減：延長保守料金（1.5～2倍）から予測可能な従量課金へ、手動作業の自動化による人的負荷軽減
  • 認証分散化の統合：現在のハイブリッド構成（オンプレミスAD + Azure AD）による運用複雑性を解消し、一元的な認証管理を実現
