10月2日の会議における技術的なやり取り

10月2日の会議で技術的なやり取り（質疑応答や議論）として行われた内容は以下の通りです。

AD (Active Directory)

  • ADが必要であるか否かについて、「ファイルサーバーの権限管理やキッティング効率化に有効である」という見解が示された。
  • 参加者から、「ファイルサーバーだけのためにADが必要か」という根本的な疑問が提示された。
  • ADを撤廃した場合、運用負荷の増大やセキュリティ劣化の懸念が大きいという指摘があった。
  • 将来的に業務を刷新・Web化する計画に合わせ、ADへの依存を低減する方向性も検討する余地がある。
  • 現段階ではADを維持することが妥当であるとの結論に至った。
  • ADはMicrosoftの製品をAWSの環境の中に配置する（AWS Managed Microsoft AD）設計であり、ライセンスは踏襲できる可能性がある。
  • ADサーバーをそのまま使う場合、ネットワーク上にサブのADサーバーが1台必要になるが、そのADサーバーの保守が1年後（11月以降）に終了するため、ADという考え方がいらないのではないかという内部的な問題提起があった。
  • AWS上でADを稼働させる設計は、構成や運用ドキュメントを開示することで、万一の際に他社や自社で引き継ぎ可能な透明性を契約上担保する方針が示された。
  • 現在ADはアジュール（Azure）の中に入っているという情報が共有された。

ファイルサーバー

  • 現在のファイルサーバーは2025年11月末で保守が終了するため、物理障害発生時に復旧やバックアップができなくなるリスクが上昇することが確認された。
  • 当面の致命的なリスクを回避するため、ファイルサーバーの保守延長の可否を確認し、実施することが決定事項となった。
  • Dropboxなどのクラウドストレージサービスでは、フォルダ単位の共有は可能だが、Windowsのファイルシステム（SMBプロトコル）とは技術的に異なるため、AccessやExcelマクロ、CSV連携といった既存業務アプリケーションが動作不能となる可能性が高いという業務断絶リスクが議論された。
    • 特に、DropboxのファイルシステムはWindowsのファイルシステムとは違うため、その上でAccessのマクロやプログラムは動作しないことが技術的な根拠として説明された。
  • ファイルサーバー上にAccessを置いて業務で使うという考え方自体が不適切ではないか、という意見が内部から出た。
  • 提案されている統合基盤では、Amazon FSx for Windows File Server（フルマネージドSMBファイルサーバー）を活用することが説明された。

ネットワーク

  • VPN関連サービスは2026年3月にベンダーのサポート体制が変更される（富士通のサポートは終了し、ソフトバンクによる継続契約は可能である）という状況が共有された。
  • 未契約や設計未整備の場合、拠点から社内接続が不可となる事態を回避するため、VPNの継続契約条件の確認と更新手続きの準備を優先的に行うことが決定された。
  • 現状のネットワーク構成について、PC約120台という規模に対してCiscoの上位L3機器が実質使われておらず、規模に合わない構成になっているため、ネットワークの簡素化（フラットな構成、L3機器の削減、回線の見直し）を行い、初期費用とランニングコストを低減すべきという要望が出た。
  • クラウド移行は、単なる「現状の置き換え」ではなく、「当面の維持・安全化」を優先し、将来的には簡素で運用容易な形へ段階的に最適化していく考え方で合意した。
  • AWSへの移行の詳細設計案と、要望されたネットワーク簡素化案（単純明快な構成）の比較資料を作成することが、今後の対応方針として示された。
  • 提案された統合基盤では、AWS Site-to-Site VPN（各拠点との安全な接続）とAWS Transit Gateway（ネットワークハブとしての機能）が利用される。

ご質問・ご指摘に対する提案理由と技術的根拠

AD (Active Directory)

ADは、Windows環境における統合認証・管理基盤であり、Kerberos認証プロトコル、グループポリシー、LDAP準拠のディレクトリサービスを提供します。提案されている統合基盤では、AWSのフルマネージドサービスである「AWS Managed Microsoft AD」を利用することで、保守負荷を削減しながらAD機能を継続します。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
ADの必要性に関する根本的な問い
「ファイルサーバーの権限管理だけのためにADが必要なのか」		 ADは単なるファイルサーバーの権限管理だけでなく、①統合認証（Kerberos）によるシングルサインオン、②グループポリシーによるPC120台の一括セキュリティ設定、③LDAP連携による他システムとの統合、④組織階層に基づくアクセス制御（RBAC）という4つの重要な機能を提供します。これらはWindows環境における標準的なエンタープライズIT基盤であり、相互に統合されています。AD廃止は技術的に現実的ではないため、継続利用を提案しています。
AD撤廃した場合の運用負荷増大とセキュリティ劣化の懸念 ADを廃止すると、各システム・サービスで個別にユーザーアカウントを管理する必要が生じます。これにより、①パスワードの使い回しや脆弱なパスワード設定のリスク増大、②人事異動時の権限変更が複数システムで必要（設定漏れリスク）、③監査証跡の収集困難、④PC設定の手動実施によるキッティング時間の増大、という運用リスクとセキュリティリスクが爆発的に増加します。そのため、AD維持を提案しています。
将来的なAD依存低減の検討と現段階での維持判断 将来的に業務システムをSaaS化すればAD依存度は低下しますが、現行の業務システム（Accessデータベース、Excelマクロ、基幹システム等）がADに依存しており、短期的な代替手段がありません。段階的アプローチとして、第1段階でADをAWS Managed Microsoft ADに移行（リフト&シフト）、第2段階でアプリケーション再設計・SaaS移行、第3段階でゼロトラストアーキテクチャへ移行という計画を提案しています。
AWS Managed Microsoft ADの採用とライセンス継続・ベンダーロックイン回避 AWS Managed Microsoft ADは、MicrosoftのAD機能をAWSのフルマネージドサービスとして提供するものです。①既存のWindowsServerライセンスをBYOL（Bring Your Own License）で継続利用できる可能性、②ドメインコントローラーのパッチ適用・バックアップ・ヘルスチェックがAWSにより自動化、③マルチAZ構成による高可用性、④構成や運用ドキュメントを公開することで他社・自社での引き継ぎが可能（ベンダーロックイン回避）というコスト削減・運用負荷削減・透明性確保のメリットがあるため、採用を提案しています。
ADサーバー保守終了に関する内部的な問題提起 オンプレミスのADサーバーを継続利用する場合、冗長構成としてサブのドメインコントローラー（DC）が最低1台必要ですが、既存DCの保守が2025年11月以降に終了します。保守終了後はハードウェア故障時の部品調達・ベンダーサポートが受けられず、業務停止に直結する重大なリスクとなります。この保守終了リスクこそがクラウド移行の契機であり、AWS Managed Microsoft ADへの移行により、ハードウェア保守からの解放・自動フェイルオーバー・ストレージの自動レプリケーションを実現できるため、提案しています。
グループポリシー（GPO）機能の重要性 ADの「グループポリシー」は、PC 120台のセキュリティ設定（パスワードポリシー、ファイアウォール、BitLocker等）、ソフトウェア配布、環境設定を組織単位（OU）ごとに集中管理する機能です。GPOがなければ、各PCに手動で個別設定が必要となり、①設定ミス・設定漏れのリスク、②セキュリティレベルの不統一、③キッティング作業時間の増大、という運用リスクとセキュリティリスクが発生します。GPOはADならではの強力な機能であり、代替手段がないため、AD維持を提案しています。
現行ADの場所（Azure）とAWSへの統合移行 現在のADがAzure上に配置されていることが判明しました。しかし、ファイルサーバー・VPN等の他のインフラがオンプレミスまたは別環境に分散しており、Azure ADとオンプレミスインフラの統合には以下の課題があります：①Azure ADとオンプレミスADの同期（AD Connect）の運用負荷、②複数クラウド環境（Azure + オンプレミス）の管理複雑化、③ネットワーク接続の複雑化（Azure ExpressRoute + オンプレミスVPN）。そのため、すべてのインフラ（AD、ファイルサーバー、VPN）をAWSに統合移行することで、①AWS Managed Microsoft ADとAmazon FSx for Windows File Serverの緊密な統合（同一VPC内での高速接続）、②AWS上での一元的な監視・バックアップ・セキュリティ管理（CloudWatch、AWS Backup、Security Hub）、③コスト最適化（Reserved Instances、Savings Plans）、④運用の簡素化（単一クラウド環境での管理）を実現できるため、提案しています。

ファイルサーバー

Windowsファイルサーバー（SMB/CIFSベース）は、ネットワーク越しにファイルを直接読み書きするプロトコル、NTFS ACLによる詳細な権限設定、レガシーアプリケーション（Access、Excelマクロ等）との互換性を提供します。現在のファイルサーバーは2025年11月末に保守が終了し、物理障害発生時の部品調達不可・ベンダーサポート終了・業務停止リスクを抱えています。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
ファイルサーバー保守終了による物理障害時のリスクと当面の対処 2025年11月末の保守終了後、ハードウェア障害（RAID構成でもコントローラー故障・複数ディスク同時故障）が発生した場合、①交換部品の提供不可、②ベンダーサポート終了によりリストア不可、③OSセキュリティパッチ提供終了、④ファームウェア脆弱性への対応不可、という可用性リスクとセキュリティリスクが顕在化します。当面の致命的なリスク回避として、保守延長の可否を速やかに確認・実施することを提案しています。並行して、クラウド移行（Amazon FSx for Windows File Server）により、ハードウェア保守からの解放を実現します。
SaaS（Dropbox等）代替の技術的な限界 Dropbox等のクラウドストレージは同期型アーキテクチャ（ローカルにダウンロード→編集→アップロード）であり、Windows環境の従来型ファイル共有（SMB/CIFSプロトコル）とは技術アーキテクチャが根本的に異なります。特に、①Access共有データベースは、複数ユーザーが同時にSMB経由でファイルを開き、レコードレベルのロック（.laccdb/.ldbファイル）を使用しますが、Dropbox同期ではファイルロック機構が正常に動作せず、データ破損・競合が発生します。②UNCパス依存（\\server\share\data.csv）のExcelマクロ・バッチスクリプトが、Dropboxのローカルフォルダパス（C:\Users\\Dropbox\...）では動作しません。③リアルタイム性が必要な業務（基幹システムがCSV出力→Excelマクロが即座に処理）において、Dropbox同期のタイムラグが業務に影響します。そのため、SMB/CIFSプロトコルをサポートするAmazon FSx for Windows File Serverを提案しています。
Access利用に関する内部的な問題提起 ファイルサーバー上にAccessデータベースを置いて共有利用する設計は、Microsoftの公式推奨からも逸脱しており、本来はSQL ServerやAccess Runtime + SharePoint等の構成が望ましいという指摘は技術的に正しいです。しかし、現状の業務フローがAccess共有DBに依存しており、アプリケーション再設計には時間とコストが必要です。段階的アプローチとして、第1段階でSMB/CIFS互換のAmazon FSx for Windows File Serverに移行し既存業務を継続、第2段階でAccessをWebアプリケーション化・データベースサーバー化という計画を提案しています。
提案されている解決策の技術的優位性（Amazon FSx for Windows File Server） Amazon FSx for Windows File Serverは、AWSのフルマネージドSMBファイルサーバーであり、①SMB/CIFSプロトコル完全サポート（既存アプリケーションとの互換性）、②NTFS ACLによる詳細な権限設定とAD統合、③マルチAZ構成による自動フェイルオーバー、④AWS Backup統合による自動バックアップ・ポイントインタイムリカバリ、⑤ストレージ容量の動的拡張（ダウンタイムなし）、⑥OSパッチ適用の自動化、⑦CloudWatch統合による監視・アラート、という技術的優位性があります。オンプレミスと比較して、可用性・セキュリティ・運用性が向上し、ハードウェア保守からの解放・コスト最適化（従量課金、初期投資削減）を実現できるため、提案しています。

ネットワーク

VPN（Virtual Private Network）は、IPsec/SSL/TLSによる暗号化通信、拠点間接続（Site-to-Site VPN）、リモートアクセス（Client VPN）を提供します。現在のVPN関連サービスは2026年3月に富士通のサポートが終了し、拠点からの社内接続が不可となる事態を回避する必要があります。

ご質問・ご指摘 なぜそのような提案をしたか（提案理由と技術的根拠）
VPNサポート終了に関する状況確認と継続契約の優先実施 VPN関連サービスは2026年3月に富士通のサポートが終了しますが、ソフトバンクによる継続契約は可能です。VPNがない場合、①営業所から本社サーバー・システムへのL3接続が不可、②ファイルサーバー（SMB）、業務システムへのアクセス不可、③代替手段（VPNなしでインターネット経由アクセス）はセキュリティリスク（平文通信、認証の脆弱性）が高い、という業務断絶リスクが発生します。そのため、VPNの継続契約条件の確認と更新手続きの準備を優先的に行うことを提案しています。並行して、AWS Site-to-Site VPNへの移行により、長期的な安定性を確保します。
ネットワーク簡素化の要望と過剰構成の指摘 現状のネットワーク構成は、PC約120台という規模に対して、Ciscoの上位L3機器（高性能なルーター/スイッチ）が導入されているものの、その高度な機能（VLAN間ルーティング、QoS制御、マルチキャスト等）が実質的に活用されておらず、規模に合わない過剰な構成になっています。これにより、①高額な機器コスト、②複雑な設定による運用負荷、③トラブルシューティングの困難化、というコストと運用の非効率が発生しています。そのため、ネットワークをフラットな構成（階層を減らす）にし、不要な高機能L3機器を削減、回線の見直し（IP-VPN集中接続化等）を行い、初期費用とランニングコストを低減することを提案しています。
移行の基本方針（当面の維持・安全化と将来の最適化） クラウド移行は単なる「現状の置き換え」ではなく、段階的アプローチを採用します。①第1段階（当面の維持・安全化）: 保守終了リスクを回避し、既存業務を継続するため、AWS Managed Microsoft AD、Amazon FSx for Windows File Server、AWS Site-to-Site VPNにリフト&シフト。②第2段階（簡素で運用容易な構成へ最適化）: ネットワーク構成の簡素化、アプリケーション再設計・SaaS移行、コスト最適化。③第3段階（ゼロトラストアーキテクチャへ移行）: VPN境界防御モデルからゼロトラスト（ID認証、デバイス認証、リソース単位の最小権限アクセス）へ。この段階的アプローチにより、業務断絶リスクを回避しながら、将来的な最適化を実現します。
提案されているAWSの構成（AWS Transit Gateway） 提案された統合基盤では、AWS Site-to-Site VPN（各拠点との安全なIPsec接続）と、AWS Transit Gateway (TGW)（ネットワークハブ機能）を利用します。TGWは、①ハブ&スポーク型アーキテクチャにより、拠点が増えても1対1接続（VPCピアリング）のように管理が複雑化しない、②ルーティングテーブルの一元管理、③Direct Connect統合（将来的な専用線接続）、④スケーラビリティ（数千の接続をサポート）、という将来の拡張性に優れたアーキテクチャです。これにより、ネットワーク構成の簡素化と将来の拡張性を両立します。
今後の比較検討（詳細設計案 vs 簡素化案） ご要望を踏まえ、AWSへの移行の詳細設計案と、ネットワーク簡素化案（単純明快な構成）の比較資料を作成します。比較項目として、①初期費用とランニングコスト（機器コスト、回線コスト、運用工数）、②可用性とセキュリティ（冗長構成、障害時RTO/RPO）、③運用負荷（設定変更、トラブルシューティング）、④将来の拡張性（拠点増加、アプリケーション追加）を定量的に評価し、最適な構成を選定します。この比較検討プロセスにより、技術的妥当性とコスト効果を明確化します。
