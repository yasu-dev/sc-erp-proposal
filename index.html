背景と課題：なぜ単純なクラウド化ではダメなのか

ミッション

オンプレミスの既存システムをAWSに移行するにあたり、セキュリティ、可用性、運用性、拡張性といった複数の要件を高いレベルで満たす最適な構成を選択すること。

現状の課題

  • オンプレミスの限界: 物理的な機器に依存するため、急な構成変更やスケールアップが困難。
  • 個別SaaSのリスク: 各SaaSでユーザー管理やセキュリティ設定がサイロ化し、組織全体での統制が難しくなる。特にアカウントの削除漏れは重大なセキュリティリスクとなる（部分最適の罠）。
  • タイムリミット: 利用中のVPNサービスが2026年に終了するという具体的な移行期限が存在する。

選択されたアーキテクチャとその技術的合理性

以下は、企業がオンプレミス環境からAWSへシステムを移行する際に、なぜ個別のSaaSツール（例：Box, Okta）を組み合わせるのではなく、「VPC（仮想プライベートクラウド）、AWS Managed Microsoft AD（Active Directory）、Amazon FSx for Windows File Server」という統合アーキテクチャを選択したのか、その技術的根拠と合理性を詳細に解説したものです。

1. ネットワーク基盤：VPCとTransit Gateway

VPC (仮想プライベートクラウド)

論理的隔離の実現

AWS内に論理的に完全に隔離されたプライベートネットワーク空間を構築。セキュリティグループやネットワークアクセスコントロールリストを駆使し、「原則禁止、許可したものだけ通信を許す」という厳格なアクセス制御を実現。これにより、万が一サーバーが侵害されても被害の拡大を防ぐ。

Transit Gateway (TGW)

複数の拠点（本社、支社）やVPCを接続する際の中央ハブとして機能。VPC同士を1対1で接続する「VPCピアリング」方式が接続先の増加で複雑化するのに対し、TGWは新しい拠点の追加もTGWへ1本接続するだけで済み、将来の拡張性に優れている。

代替案（VPC Peering）の不採用理由

VPC Peeringは1対1の接続であり、拠点数が増えると設定が指数関数的に増加し、管理が破綻します。10拠点で45個、20拠点で190個の接続を管理する必要があり、運用負荷が指数関数的に増加します。

2. 中核サービス：なぜADとWindowsファイルサーバーをAWS上で再構築するのか

この構成の核心は、SaaSでは代替が難しい「既存業務との完全な互換性」と「組織全体の一元的な統制」を最優先した点にあります。

Amazon FSx for Windows File Server

互換性

オンプレミスのWindowsファイルサーバーで利用されている複雑なアクセス権限設定（NTFS ACL）、シャドウコピー（ユーザーによる自己復元機能）、SMBプロトコルといった機能を完全にサポート。これにより、利用者は移行を意識することなく、既存の業務アプリケーションも改修なしでそのまま利用できる可能性が高い。

SaaSの限界

  • 一般的なファイル共有SaaSでは、日本企業特有の部署や役職に基づいた複雑な権限設定の再現が困難
  • 既存の業務アプリが利用するSMBプロトコルに対応していないため、アプリの大規模な改修が必要となる
  • 監査証跡の不十分さ：取得項目や保管期間に制限があり、J-SOXやISMSなどの内部統制監査で要求される詳細な証跡を完全に満たすことが困難

AWS Managed Microsoft AD (Active Directory)

信頼の基盤

単なる認証システムではなく、ITインフラ全体の信頼と統制の中心（トラストアンカー）として位置づけられています。

具体的な役割

1. 統合認証 (SSO)

一度のログインで様々な連携システムにアクセス可能にし、利便性とセキュリティを向上させる。

2. クライアントPCの強制管理 (GPO)

パスワードポリシー、USBメモリの使用禁止、セキュリティソフトの一斉導入など、全社員のPCに統一されたセキュリティ設定を強制適用できる。これはADならではの強力な機能。

3. ライフサイクル管理

社員の退職時にADアカウントを無効化するだけで、連携する全てのシステムへのアクセスを即座に遮断でき、情報漏洩リスクを根本から低減する。

3. マネージドサービスによる高可用性と運用効率化

高可用性

Managed ADとFSxは、物理的に独立した複数のデータセンター（マルチAZ）に自動でシステムを分散配置する構成が標準で提供される。これにより、一つのデータセンターで大規模災害が発生してもサービスは停止せず、オンプレミスでは実現困難な99.9%以上の高い可用性を確保できる。

運用負荷の軽減

OSのパッチ適用、データのバックアップといった煩雑な運用作業はAWS側で自動的に行われるため、社内のIT担当者は本来注力すべき業務に集中できる。

AWS統合アーキテクチャの全体的な優位性

運用監視の一元化

ネットワーク、認証、ファイルサーバーなど、各コンポーネントのログや監視データをAWSの「CloudWatch」に集約。システム全体の状況を一つの画面で把握し、問題発生時の原因特定を迅速化できる。

コンプライアンスと監査対応

「CloudTrail」やADのログを改ざん防止機能のあるストレージ（S3）に長期保管することで、J-SOX法やISMSなどの厳しい監査要件に、信頼性の高い証跡をもって対応できる。

コストの最適化

初期費用が安いイメージのあるSaaSだが、同等の機能（ユーザー数、容量、セキュリティ要件）を複数のSaaSで満たそうとすると、長期的な総所有コスト（TCO）ではAWSに優位性がある。

SaaS組み合わせ型の問題点

運用と障害切り分けの困難さ

  • 複数のSaaSを組み合わせた環境で障害が発生した場合、問題の切り分けが非常に困難
  • 各ベンダーのサポートに問い合わせる必要があり、責任の所在が不明確
  • 解決までの時間（MTTR）の長期化

セキュリティと監査の不整合

  • 各SaaSは独自のログ形式と保持ポリシーを持つ
  • 横断的な監査やインシデント追跡が困難
  • 退職者のアクセス権が各SaaSに残留するリスク

慎重な移行計画

段階的移行アプローチ

  • リスクを最小化するため、一括移行ではなく段階的に進める計画
  • データ移行には「AWS DataSync」を使い、変更があった差分データのみを同期させることで、最終的なシステム切り替え時の停止時間を最短にする
  • 移行前には入念なリハーサルを行い、万が一に備えた詳細な切り戻し（ロールバック）計画も準備する

データ移行の詳細

  • Phase 1: アーカイブデータの移行（週末実施）
  • Phase 2: 部門別データの順次移行
  • Phase 3: アクティブデータの最終同期
  • カットオーバー: DNSレコードの切り替えとDFS名前空間によるシームレスな切り替え

🎯 結論

このAWS統合アーキテクチャは、単に機能をクラウドに移行するだけでなく、以下の5つの核心的な価値を実現するための選択です。

1

統合されたセキュリティ

VPCによる境界防御とADによる統合認証で、攻撃面を最小化し内部統制を強化

2

保証された可用性

マルチAZ構成とマネージドサービスで99.9%以上の稼働率を保証

3

効率化された運用

AWSネイティブツールによる自動化で人的リソースとミスを削減

4

将来の拡張性

業界標準のAPIとプロトコルで柔軟なシステム拡張を可能に

5

全体最適化されたコスト

従量課金制とSaaSの適材適所活用でTCOを最小化

目先の導入の容易さや個別の機能の魅力に囚われる「部分最適」に陥ることなく、既存業務との継続性や組織全体の統制といった、ビジネスの根幹を支えるための「全体最適」を追求した結果導き出された、現時点で最も合理的かつ信頼性の高い技術的選択肢です。
